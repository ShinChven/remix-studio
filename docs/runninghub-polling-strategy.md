# RunningHub Polling & Retry Strategy

Unlike synchronous providers (like OpenAI or Google Vertex AI) which hold the HTTP connection open until the image is returned, **RunningHub** operates asynchronously. When a generation request is sent, RunningHub immediately returns a `taskId`, and the client must poll their API to check the status and retrieve the final image.

To support this without blocking the worker queue, the `QueueManager` uses a **detached polling strategy**.

## 1. Initial Generation Request
When the `QueueManager` executes a job using the `RunningHubGenerator`:
1. `generator.generate()` is called.
2. The generator submits the prompt to RunningHub and receives a `taskId`.
3. It immediately returns a `GenerateResult` with `status: 'processing'` and the `taskId`.
4. The `QueueManager` saves this `taskId` to the job in the database and **stops blocking the queue**. The job remains in the `'processing'` state.

## 2. Detached Polling
A separate interval-based detached poller runs every 30 seconds (`QueueManager.pollDetachedTasks()`).
1. It queries the database for all jobs with `status: 'processing'` that have a `taskId`.
2. For each job, it calls `generator.checkStatus(taskId)` to check RunningHub's progress.
3. If RunningHub is still processing, the job is ignored until the next tick.
4. If RunningHub returns `'completed'` along with the `imageBytes`, the poller hands the bytes over to `processCompletedImage()` for local S3 storage and thumbnail generation.

## 3. Handling Local Processing Failures
If an error occurs *after* the image is successfully generated by RunningHub (e.g., the local S3 storage throws a quota exceeded error during `processCompletedImage`), the system must **not** discard the `taskId`.

**Crucial Error Handling Rule:**
When a local processing error occurs, the job is updated to `status: 'failed'` to notify the user, but the **`taskId` is deliberately preserved in the database**.

## 4. Retry Behavior
If the user resolves the local issue (e.g., deleting old images to free up storage quota) and clicks "Retry" on the failed job:
1. The frontend sets the job back to `status: 'pending'`.
2. The `QueueManager` picks it up again in `executeJob()`.
3. Because the job already has a `taskId`, the `QueueManager` **skips calling `generator.generate()`**.
4. It simply flips the status back to `'processing'` and immediately hands the job back to the detached poller.
5. The detached poller picks it up on the next tick, realizes RunningHub has already finished the task, and attempts to save the image locally again.

This strategy ensures that temporary local failures do not result in double-billing or wasting credits on RunningHub by re-submitting the same prompt.
